
# CODING_STANDARDS.md

# FlowSense Engineering & Coding Standards

## Purpose
These standards are mandatory for all code generated by Claude Code, Codex, or human contributors. Prioritize maintainability, readability, modularity, and performance over shortcuts.

---

# Core Principles

- SOLID principles
- DRY (Don't Repeat Yourself)
- KISS (Keep It Simple)
- Composition over inheritance
- Explicit code over clever code
- Local-first architecture

Never sacrifice code quality for speed.

# Folder Organization

Each module should have a single responsibility.

Avoid files larger than ~400 lines when practical.
Split reusable logic into services, hooks, utilities, or components.

---

# React Standards

- Functional components only
- Strict TypeScript
- No `any` unless unavoidable and documented
- Reusable hooks for shared logic
- Keep components focused and testable
- Memoize expensive computations where beneficial

Naming:
- PascalCase for components
- camelCase for variables/functions
- UPPER_SNAKE_CASE for constants

---

# Backend Standards

- Thin routes
- Business logic in services
- Database access isolated
- Dependency injection where appropriate
- Validate every request with Pydantic

Never access SQLite directly from route handlers.

---

# Database Standards

- Versioned migrations
- Foreign keys enabled
- Indexed frequently queried columns
- Never modify historical migrations

---

# Error Handling

- Never expose stack traces to users
- Log structured errors
- Return consistent JSON responses

Example:
{
  "success": false,
  "error": "Validation Error"
}

---

# Logging

Use structured logging.

Include:
- timestamp
- module
- severity
- message

Never log sensitive user activity unnecessarily.

---

# Security

- No cloud storage in MVP
- Sanitize all inputs
- Escape user-provided text
- Principle of least privilege

---

# Performance Targets

- Startup < 3 seconds
- Dashboard < 500 ms
- Activity ingestion < 25 ms
- Memory usage < 300 MB under normal operation

---

# Testing

Frontend
- Component tests
- Hook tests

Backend
- Unit tests
- Integration tests
- API tests

Algorithm
- Deterministic tests
- Edge cases
- Performance benchmarks

Target:
>80% code coverage.

---

# Git Workflow

Branch naming:
feature/<name>
fix/<name>
docs/<name>

Commit format:
feat:
fix:
docs:
refactor:
test:
chore:

Commit after each completed milestone.

Push using the GitHub account already authenticated on the development machine.

---

# Documentation

Every module should include:
- Purpose
- Inputs
- Outputs
- Dependencies
- Limitations

Public functions require docstrings.

---

# Code Review Checklist

- Readable
- Typed
- Tested
- Documented
- No duplicated logic
- No dead code
- No debug statements

---

# Accessibility

- Keyboard accessible
- WCAG AA contrast
- Semantic HTML
- ARIA labels where appropriate

---

# Dependency Rules

Prefer stable, well-maintained libraries.

Avoid unnecessary dependencies.

---

# Definition of Done

A feature is complete only if:
- Implemented
- Tested
- Documented
- Linted
- Type-safe
- Reviewed
- Committed
- Pushed
